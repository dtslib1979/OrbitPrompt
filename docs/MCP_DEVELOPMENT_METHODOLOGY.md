> **작성:** 2026-06-04  
> **출처:** 박씨 × LLM 대화 기반 MCP 개발 실제 사례  
> **핵심:** "에이전트로 MCP를 만들고, MCP로 LLM을 족쇄 채운다"  
> **증명:** Election MCP direction_acc 100% (8/8)

## 0. 한 줄 정의

> **인간 사고 ↔ LLM 대화 → 에이전트가 MCP 코드/스펙 생성 → 완성된 MCP가 다시 LLM을 로데이터 생성기로 제한**

---

## 1. 기존 방식과의 차이

| 구분 | 기존 MCP 개발 | PARKSY MCP 개발법 |
|------|-------------|-------------------|
| 시작점 | "어떤 도구를 만들까" | **"내 생각은 이렇다" — 대화** |
| 설계 주체 | 개발자 혼자 설계 | **인간 + LLM 대화로 공동 설계** |
| 구현 | 개발자가 코드 작성 | **에이전트(LLM)가 server.py 생성** |
| LLM의 역할 | 도구를 사용하는 주체 | **MCP 안에서 로데이터 생성기로 제한** |
| 족쇄 | 없음 (LLM 자유롭게 응답) | **Python 래퍼 + delta layer로 고정** |
| 검증 | 수동 테스트 | **validate_fixtures 자동 측정** |
| 문서화 | 별도 작성 필요 | **MCP가 명세서/설명서까지 생성** |
| 전시 | 없음 | **갤러리에 자동 전시** |

## 2. 5단계 개발 프로세스

### Stage 1: 도메인 고르기

무엇을 모델링할 것인가. 박씨의 경우 = 정치 선거 예측.

```
기준:
- 본인이 10년+ 경험을 가진 도메인
- 반복되는 판단 패턴이 있는 분야
- "이건 구조가 있다"고 느껴지는 영역
```

### Stage 2: 대화로 사고를 꺼낸다

LLM과 대화하면서 네 머릿속 구조를 언어로 뽑아낸다.

```
방법:
- "도전자는 130% 해야 이긴다"
- "TK/호남은 구조적 불가능"
- "이재명은 장르가 아니라 예외"
- "서울은 부정부패 있어도 버티는 구조"

이런 문장들이 나오면 → 변수/규칙/제약 조건으로 정리
```

**산출물:** g/d/i/o/R 변수 정의, delta layer 규칙, 아키타입 분류

### Stage 3: 에이전트가 MCP 서버를 만든다

LLM에게 "이 규칙들로 MCP 서버를 만들어줘"라고 시킨다.

```
LLM이 생성하는 것:
├── server.py          # FastMCP 서버 (385줄)
├── config/*.json      # 설정 파일 (hero/weights/frame/delta)
├── tools/list         # 도구 목록 (10개)
├── tools/call         # 각 도구 실행 로직
├── docs/spec.md       # 기술 명세서
├── docs/manual.md     # 사용 설명서
└── boards/gallery.html # 전시 갤러리
```

**핵심:** "네 생각을 Python 래퍼로 씌워서 도망 못 가게 족쇄 채운다."

### Stage 4: 족쇄 조정 — delta layer 튜닝

모델이 예측한 결과를 보고 "이상하다" 싶은 부분을 delta layer로 추가한다.

```
예시:
- "김부겸도 못 이긴 TK" → regional_impossibility 레이어
- "김어준식 정신승리" → challenger_130 레이어
- "민주당은 연합정당" → democratic_coalition_risk 레이어
```

**결과:** config/*.json만 수정하면 모델 업데이트 완료. 재학습 0회.

### Stage 5: 갤러리에 전시 + STT 연결

완성된 MCP의 결과물을 갤러리에 전시하고, 음성으로 입력할 수 있게 연결한다.

```
최종 상태:
🎤 말한다 → 📝 STT 기록 → 🧠 LLM 해석 → ⚡ MCP 실행 → 🌐 갤러리 전시
```

## 3. 이 방법론의 본질

> **"나는 MCP로 에이전트를 만든 게 아니라, 에이전트로 MCP를 만들었다."**

| 단계 | 누가 | 무엇을 |
|------|------|--------|
| 1 | 인간 | LLM과 대화하며 사고를 구조화 |
| 2 | LLM | 대화 내용을 변수/규칙으로 정리 |
| 3 | 에이전트 | server.py + config + docs 생성 |
| 4 | 인간 | delta layer로 족쇄 조정 |
| 5 | LLM | MCP 안에서 로데이터만 생성 (족쇄 상태) |

---

## 4. 기존 방법론과의 차별점

| 기존 AI 개발 | PARKSY 방법론 |
|-------------|--------------|
| "LLM을 파인튜닝해서 내 생각을 학습시킨다" | **"내 생각을 Python 래퍼로 감싸서 LLM을 제어한다"** |
| LLM이 블랙박스라 왜 그런 결과가 나왔는지 모름 | **JSON 입출력으로 모든 추적 가능** |
| 업데이트하려면 재학습 | **config 수정으로 즉시 업데이트** |
| 검증 불가능 | **direction_acc 100% 측정 가능** |
| 하나의 모델에 모든 지식이 섞임 | **MCP 단위로 모듈화** |

## 5. 이 방법론이 가능했던 이유 (AI 시대)

AI 시대가 아니었다면 불가능했던 것:

1. **STT로 말하면 LLM이 알아듣고 MCP를 생성** — AI 없으면 성립 안 함
2. **385줄로 10개 도구를 가진 서버를 혼자 만든다는 건 불가능** — 에이전트가 코드를 생성
3. **"파인튜닝 없이 사고방식을 복제한다"는 발상 자체** — AI가 없었다면 태어나지도 않음

근데 AI 시대라고 다 이렇게 하는 건 아니다.
ChatGPT 쓰는 99%의 사람은 "남이 만든 도구 사용법을 익히는 중"이고,
박씨는 "자기 사고방식을 복제하는 MCP를 만드는 중"이다.
이 차이는 운이 아니라 판단이다.

## 8. 결론

> **"에이전트로 MCP를 만들고, MCP로 LLM을 족쇄 채운다.**
> **이 전 과정을 회의록+대화형 설계로 끝낸 게 PARKSY MCP 개발법이다.**

| 기존 개발법 | PARKSY 개발법 |
|-----------|--------------|
| 요구사항 정의 → 설계 → 구현 → 테스트 → 배포 | **대화 → 구조화 → MCP 생성 → 족쇄 조정 → 전시** |
| 개발자가 모든 걸 한다 | **인간이 방향을 정하고, 에이전트가 만든다** |
| LLM은 도구 사용자 | **LLM은 로데이터 생성기 (족쇄 상태)** |

**이 방법론은 세상에 존재하지 않았다. 박씨가 처음 만들었다.**
