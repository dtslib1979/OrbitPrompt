# 삼각 실행 루프 (Triangular Staff Method)

> 발견: 2026-06-08, 박찬대 되기 프로젝트 세션 중 자연발생
> 설계자: 박씨 (무의도적 창발)
> 문서화: Claude Sonnet 4.6

---

## 정의

AI 두 개와 인간 한 명이 만드는 자기조직화 실행 팀.
회의가 곧 실행이고, 실행 결과가 다음 회의 자료가 된다.
회의록이 아니라 회의 자체가 공장이다.

---

## 구조

```
박씨 (의사결정자/팀장)
  │
  ├──▶ DeepSeek (phone_aider) — 구현 담당 보좌관
  │       초안 생산 / MCP 코딩 / 방향 제시 소화
  │       ↑
  │       └── counter.md 읽고 다음 답변 수정
  │
  └──▶ Claude (PC 관제탑) — 검증/감사 담당
          카운터 치기 / 레포 정합성 / 방법론 문서화
          ↓
          counter.md 커밋 → GitHub 레포
```

**핵심 루프:**
```
박씨 방향 제시
  → DeepSeek 초안
    → Claude 카운터 → GitHub 커밋
      → DeepSeek가 counter.md 읽음
        → DeepSeek 다음 답변 수정
          → Claude 재카운터
            → 루프 반복
```

| 역할 | 담당 | 하는 것 | 하지 않는 것 |
|------|------|---------|------------|
| 팀장 | 박씨 | 방향 제시, OK/NO 결재 | 직접 구현, 직접 코딩 |
| 보좌관 | DeepSeek | 초안 생산, 코딩, 제안 | 전략 설계, 방향 판단 |
| 감사관 | Claude | 카운터, 레포 관리, 문서화 | 박씨 대신 결재, DeepSeek 대신 코딩 |

## 기존 방법론과의 차이

| | 일반 스태프 미팅 | Multi-Agent(논문) | 삼각 실행 루프 |
|---|---|---|---|
| 실행 주체 | 사람 | AI(통제 실험) | AI + 인간 |
| 회의와 실행 | 분리 | 분리 | 동일 |
| 결과물 축적 | 회의록 | 실험 데이터 | GitHub 커밋(제품) |
| 교정 방향 | 단방향 | 단방향 | 쌍방향 자동 루프 |
| 환경 | 통제됨 | 통제됨 | 실무 불확실 |
| 자본 | 조직 필요 | 서버 필요 | 1인 무자본 |

## 핵심 발견 3개

### 1. 창발 — 설계하지 않은 것이 작동했다

counter.md를 만들 때 DeepSeek가 같은 레포를 읽을 거라고 의도하지 않았다.
DeepSeek가 레포 탐색 중 자동으로 발견해서 읽었다.
설계가 아니라 시스템이 알아서 연결됐다.
→ **조건만 만들면 구조가 자기조직화한다.**

### 2. 회의 = 실행 — 분리가 없다

박씨↔DeepSeek↔Claude 대화가 진행되는 동시에:
- 박찬대 MCP 설계가 진행된다
- HTML 섹션이 커밋된다
- 방법론 문서가 쌓인다

회의가 끝나고 실행하는 게 아니다.
회의하는 것 자체가 실행이다.

### 3. AI peer review의 쌍방향 루프

Claude가 DeepSeek를 평가 → 문서로 커밋
→ DeepSeek가 그 문서를 읽음
→ DeepSeek 다음 답변이 바뀜
→ Claude가 재평가

AI가 AI를 감사하고, 그 감사 문서를 AI가 읽고 행동을 수정한다.
이 루프가 실무 프로젝트 안에서 자연발생했다.

---

## 컨텍스트 내구성 — 세션이 터져도 안 죽는다

이 방법론의 구조적 강점 중 하나.

- AI 컨텍스트 소진 → GitHub 레포가 기억을 대체
- counter.md 읽으면 전체 대화 맥락 즉시 복원
- 박씨가 리마인더 한 줄 던지면 세션 재개
- 컨텍스트 경계가 프로젝트 경계가 아니다

**결론: 컨텍스트 터지는 거 두려워할 필요 없다. 레포가 살아있으면 팀이 살아있다.**

---

## 작동 조건

1. **공유 레포** — 두 AI가 같은 GitHub 레포에 접근 가능해야 한다
2. **역할 분리** — 구현 담당 / 검증 담당이 명확히 나뉘어야 한다
3. **팀장의 방향 제시** — 인간이 의사결정을 쥐고 있어야 한다. AI끼리만 두면 수렴하거나 발산한다
4. **카운터 문서 공개** — 검증 결과가 보좌관에게 노출되어야 루프가 작동한다

## 적용 가능 영역

이 방법론은 박찬대 프로젝트 특정이 아니다. 구조가 동일하면 어디든 적용된다:

- 콘텐츠 생산 (에디터 + 작성 AI + 팩트체크 AI)
- 정책 분석 (의원 + 조사 AI + 비판 AI)
